# 8ball — DOCTRINE.md

> One page. Read every release. Edit only deliberately.

This document is the constitution of `8ball`. The codebase obeys it. PRs that contradict it require an explicit doctrine amendment in the same change.

## §1. What this is

A fixed designed deck that knows you. Enter your name and date of birth once. Shake.

**Current free/paid composition (authoritative) — read this before the lineage prose below.** The free surface is **five** coordinates from the date of birth (tarot birth card · sun · public animal (year) · life path · catalog numeral), rendered on a constant compartment **specimen sheet** where higher-tier coordinates appear as sealed cells (§1.D v0.37); three paid rungs (t1/t2/t3) open the rest. The current per-surface authority is the §1.x clause family — §1.A rising · §1.B numerology · §1.C birth card · §1.D tier ladder · §1.E provenance · §1.F atlas+density · §1.G coordinate meanings · §1.H t3 written-entry rotation · §1.I registry + concordance. The two paragraphs immediately below describe the original v0.2.3 surface and are retained as lineage per L17; where they disagree with §1.A–§1.I, the clauses win.

The product calculates ten baseline calibrated coordinates from (name, DOB): sun sign, Chinese five-element, public animal (year-pillar), private animal (month-pillar), life path, expression/name number, personality, maturity, soul urge, and birthday. The two animals pair on a single line via an equilibrium arrow (`⇌`); three of the numerology coordinates (life path, expression/name number, soul urge) render as a text triplet on line 4 of the result card. The (sun sign, public animal) pair drives a 144-card catalog index — computed positionally in `core/engine.js` (sun-row × 12 + animal-col + 1, rendered as roman numeral). Life path drives bracket resolution (low/mid/high) within a card cell via `resolveBracket`, not the catalog index. The catalog index is the only card-derived field surfaced.

As of v0.2.3 the result card surfaces seven of the ten baseline coordinates (chinese five-element, sun, public animal, private animal, life path, expression/name number, soul urge), optional rising sign, and the catalog index. The remaining three baseline coordinates (personality, birthday, maturity) are computed on buildProfile but reserved for v0.3.0+ paid surfacing — see §1.B. The locked render path ships symbols only — no interpretation, no per-symbol explanation — and is the canonical free surface. The unlocked render path (gated by paid credits per §1 v0.22 amendment / §4.B / §5.C) surfaces the card-content layer (name, type, habit, note per cell) from `content/cards.v1.full.js`; private authoring source is preserved at `~/dev/8ball-private/cards.v1.full.js`.

The voice is declarative-observational, framed in strengths and weaknesses. Cards openly reference the symbol systems they draw from — sun sign, Chinese five-element, Chinese zodiac animals (year and month pillars), numerology life path, name number, soul urge — and name them as such.

**§1.A. Rising sign — surface coordinate, not a driver.**

As of v0.2.1 the result card surfaces an eighth coordinate when the operator provides birth time, country, latitude, and longitude: the rising sign (astronomical ascendant). Rising sign is computed from `(year, month, day, hour, minute, utcOffsetMinutes, lat, lng)` via standard ascendant math (Meeus); see `core/rising.js` for the exact algorithm. It is rendered on line 2 of the result card paired with the sun sign as `${sunSign} ↑ ${risingSign}`.

Rising sign is **surface-only**: it does not enter `getCard`, `resolveBracket`, or any catalog computation. The catalog driver remains `(sunSign, animal)` per §1. Rising adds a coordinate to the visual surface; it does not branch the card layer.

When required birth-time or location fields are missing, rising sign is `undefined` and line 2 falls back to the v0.2.0 bare-sun-sign render. Existing v0.2.0 profiles in localStorage continue to work without modification.

**v0.21 amendment (v0.2.7.2).** Birthplace input is city-level via `core/cities.js` — a 53,308-entry pop≥7500 subset of GeoNames cities5000, lazy-loaded from `assets/cities.json` per §5 (v0.21 same-origin lazy-load permission). Selecting a city atomically sets `tz` (IANA timezone), `lat`, `lng`, and country code on the form state. The country-code payload shape from v0.2.1 is retained for backward compatibility with stored v0.2.1–v0.2.7.1 profiles, but the UI no longer surfaces country selection.

Rising-sign math is now DST-aware and historical-tz-aware. The `computeRising({...tz...})` API in `core/rising.js` resolves the UTC offset for the (year, month, day, hour, minute) wall-time input via `Intl.DateTimeFormat` with `{ timeZone: tz, timeZoneName: 'longOffset' }` — gives correct offset including DST transitions and historical-rule changes (e.g. Indiana pre-2006 no-DST, Russia post-2014 permanent UTC+3, US summer DST). As of v0.5.2, stored legacy country-code profiles first resolve the country/zone code to a representative IANA timezone, then use the same `computeRising` path as fresh city profiles. Legacy `getRisingSign(..., utcOffsetMinutes, ...)` is retained only as a low-level fixed-offset helper; both APIs are polar-safe.

Polar latitudes (|lat| > 66.5°, strictly past the polar circles) return `null` from both rising APIs. The horizon-plane geometry degenerates at the poles and a rising sign is not astrologically meaningful inside the polar circle. The UI surfaces "rising unavailable at this latitude" at city-selection time when |lat| > 66.5°. Boundary cases (|lat| == 66.5°) return a valid sign — the rule is strict greater-than.

Pre-1970 dates: `Intl.DateTimeFormat`'s historical-tz coverage falls back to LMT (Local Mean Time) approximation for some timezones in pre-tzdata-coverage periods. This is acknowledged as a precision tradeoff; the about-modal disclosure copy lands at v0.2.8 (calculator-framing rewrite cycle).

**§1.C. Tarot birth card — surface coordinate, not a driver (v0.5.0).**

As of v0.5.0 the result card surfaces a tarot birth card: a Major Arcana coordinate computed from the DOB alone, rendered as roman numeral + lowercase name (e.g. `XXI · the world`). It is the **lead row** of the free card, above the five-element coordinate. Computed in `core/birthcard.js` (`getBirthCard`), carried as the additive `birthCard` field on `buildProfile`.

This is a **free-surface coordinate**, inside the §1 symbols-only invariant — a labeled coordinate (numeral + arcana name), never interpretation. No per-card meaning text, no per-arcana explanation; that register stays the paid card-content layer (§1 v0.22). **Superseded in part by §1.G (v0.44):** this coordinate (and sun / public animal / life path, the other three free-tier VALUES) may now surface a static, tradition-cited meaning entry on tap — citation of what a named tradition says, not the interpretation this clause originally reserved for the paid layer. The distinction §1.G draws is citation-vs-interpretation, not free-vs-paid; the paid card-content layer (name/type/habit/note for the compound catalog reading) remains untouched and is not given away by §1.G. The clause adds a coordinate to the visual surface in the same way §1.A's rising sign does — it does **not** enter `getCard`, `resolveBracket`, or any catalog computation; the catalog driver remains `(sunSign, animal)` per §1.

Reduction tradition is **digit-sum**: sum every digit of the DOB, reduce by repeated digit-sum until ≤ 22. This is distinct from the life-path reduction in `core/profile.js` (which stops at master numbers 11/22/33 and floors at a single digit); the birth-card reduction floors at ≤ 22 with no master-number stop, because the target range is the 22 Major Arcana. A reduced value of 22 maps to `0 · the fool`, guaranteeing every DOB resolves to a real arcana (the out-of-range `XXII` never renders). Both choices are operator-locked (see journal); codified by `tests/birthcard.test.js`.

Because it is a free coordinate, the birth card is included in the §5.D share surface — the shared PNG and the on-screen free card carry the same five coordinates. It carries no PII (numeral + arcana name only, no name/DOB) and no paid-content layer, so the §5.D invariants (a)/(b) hold unchanged.

**§1.I. Registry + Concordance — finite post-calculation relations (v0.51, controller override 2026-07-19).** Previous Readings MAY let the user select exactly two valid saved entries and open a separate Concordance screen. Both stored input payloads are recomputed independently through the unchanged `profileFromPayload` → `buildProfile` path; only after both calculated profiles exist does `ui/concordance.js` look up matching axes. The calculation core, catalog driver, bracket resolution, card output, tries, and credits are not inputs to relation lookup and are not mutated; the stored device tier is read once, solely to decide whether the five-element axis is included at all (§1.D entitlement), and never alters any axis's status or output. The compared axes are sun sign × sun sign, public animal × public animal, life path × life path, birth card × birth card, and five-element × five-element only when the device tier entitles the element coordinate.

**Register law.** Every axis emits exactly one textual status from `registered | adjacent | unfiled`, plus the two coordinate values, relation text, named registry, filing citation, and the qualifier `recorded, not certified.` `registered` means a named finite registry contains that relation; `adjacent` means two records may be juxtaposed but no shared registry is claimed; `unfiled` means the checked registry contains no named relation for that pair. The MVP's matching-axis lookups do not fabricate an `adjacent` relation: unsupported or same-value pairs are `unfiled`. The v1 registered inventory is finite and pinned by `tests/concordance.test.js`: 66 distinct western sign-distance pairs, 33 named earthly-branch animal pairs, 10 distinct wuxing phase pairs with direction, 3 Pythagorean master-reduction links, and 21 adjacent project-trump sequence pairs. Static data lives in immutable `content/concordance.v1.js`; a changed registry ships as a new versioned file.

**Anti-oracle boundary.** Concordance is structural citation, not interpersonal interpretation. It MUST NOT emit a compatibility percentage, harmony score, ranking, prediction, advice, diagnosis, soulmate claim, recommendation, or invented synthesis across traditions. Local saved titles may label the two on-screen columns; the screen does not show DOB. No comparison share artifact ships in this MVP. ## §3. The calculation contract

`core/profile.js` is the calculation core. Changes fall into two categories:

**Breaking changes** — modify existing outputs (sunSign, animal, lifePath, nameNumber, chineseElement, soulUrge for any input). The contract:

1. Add or update fixtures in `tests/fixtures.json` reflecting the intended new behavior.
2. Update the algorithm.
3. Run `npm test`. All cases must pass.
4. Bump the calc-version note at the bottom of this file.

**Additive changes** — add new outputs without modifying existing ones (e.g. new functions exported, new fields on the `buildProfile` return value). The contract:

1. Add direct unit tests for the new outputs in `tests/profile.test.js`.
2. Run `npm test`. All cases must pass, including the new tests.
3. Existing `tests/fixtures.json` cases must remain byte-identical (no silent drift).
4. Calc-version stays the same major (additive doesn't break v1 consumers).

Algorithm versions documented:

- **calc v2** (v0.2.7.1) — Real lunar new year table for year-pillar cusps (year-pillar animal + 5-element pillar). Real solar-term table for month-pillar (`innerAnimal`). Both tables span 1900–2100, evaluated at date-precision in canonical Asia/Shanghai timezone (UTC+8); DOB given as `YYYY-MM-DD` is treated as the truth — no timezone conversion of user input. Lunar new year dates and the 12 month-start solar-term dates are computed in JS via Meeus astronomical algorithms (Chapter 25 sun position, Chapter 49 lunar phases with periodic + planetary corrections), mirroring the `core/rising.js` Meeus pattern; results match Hong Kong Observatory canonical tables byte-exact at the sanity-lock checkpoints in `tests/profile.test.js`. Zero runtime deps, zero devDeps, zero build step. The v1 Feb-4 fixed-cutoff approximation is retired. **`getInnerAnimal` signature changed from `(month, day)` to `(year, month, day)`** as a breaking calc change. All existing fixtures other than the v1-bug-locking fixture #6 (CNY Feb 4 cutoff) remain byte-identical; fixture #6 is intentionally re-spec'd because it was locking the bug being fixed.
- **calc v1** (initial through v0.2.7) — Pythagorean life path with master numbers 11/22/33 preserved at the final reduce. Western tropical sun signs at standard cusps. Chinese zodiac with Feb 4 cutoff approximation (retired in calc v2). Additively extended at v0.2.0 with: chinese five-element (2-year cycle, 1924 anchor), public/private animal split (year-pillar already present at `animal`; month-pillar added as `innerAnimal` via solar-term cutoff approximations), soul urge (vowel-only Pythagorean, masters preserved), and unreduced sums (lifePathSum, nameNumberSum, soulUrgeSum) exported for potential future surfacing. All additions covered by direct unit tests in `tests/profile.test.js` and `tests/rising.test.js`; existing fixtures unchanged.

If a fixture changes silently, the test gate catches it. If a test changes silently, code review catches it. If both change in the same commit without a doctrine note, the reviewer rejects.

The product persists only user-entered profile fields, in `localStorage` only, on the user's own device:

- `name` (string)
- `dob` (ISO date YYYY-MM-DD)
- optional `time` (HH:MM string, 24-hour)
- optional `country` (country/zone entry code from `core/countries.js`)
- optional `lat` (decimal latitude)
- optional `lng` (decimal longitude)
- optional `city` (city display name from `assets/cities.json` — user's birthplace selection. Added v0.2.7.2.)
- optional `cc` (ISO 3166-1 alpha-2 country code, paired with `city` for display rehydration. Added v0.2.7.2.)
- optional `tz` (IANA timezone string — drives DST-aware rising-sign computation per §1.A v0.21. Added v0.2.7.2.)
- `eight_ball_labels_revealed_v1` — separate string flag (`'true'` or `'false'`) tracking whether the user has chosen to reveal the symbol-name labels on the result card. UI preference only; independent of the profile payload. Added v0.2.7.
- `eight_ball_tries_used_v1` — integer (serialized as string). Monotonic counter incrementing on every new-pair shake regardless of paid state. Read against the constant `3` for the §4.B cap check. Added v0.3.0.
- `eight_ball_credits_v1` — integer (serialized as string). `+= 3` on each `?paid=t1` handler invocation; decrements by 1 on each new-pair shake when `> 0`. Hits zero → next new-pair triggers paywall or locked render depending on `tries_used`. Non-negative invariant. Added v0.3.0.
- `eight_ball_pending_profile_v1` — JSON-stringified profile object, transient. Written when the paywall modal opens (both Path A new-pair shake and Path B lock-tap on already-rendered card per §5.C). Consumed on the `?paid=t1` return: one credit decremented, profile saved to `eight_ball_profile_v1`, key deleted. Lifetime is one round-trip; either absent or present-and-valid. Added v0.3.0.
- `eight_ball_tier_v1` — string tier flag (`'t1' | 't2' | 't3'`), the highest rung purchased per §1.D; absent = free. Monotonic — written only by the `?paid=t1|t2|t3` return handler as max(current, purchased) by ladder rank, plus the one-time §1.D legacy-grandfather write (credits with no tier key persist as `'t3'` on first render); never downgrades. - `eight_ball_facet_index_v1` — integer string (`'0' | '1' | '2'`) naming the currently visible v1 note position for the t3 written entry, never the next position. Profile-free and PII-free. New profile and consumed pending-profile return reset it to the life-path anchor; reload and same-profile submission preserve it; `forget` clears it. It changes only alongside a funded explicit t3 `flip again`; a zero-credit paywall opening leaves it unchanged. Added v0.49 (§1.H).
- `eight_ball_saved_readings_v1` — JSON array for the user-initiated Saved Readings archive. Each entry contains local metadata (`id`, user-editable `title`, ISO `savedAt`) plus a `profile` object restricted to the same user-entered reconstruction fields allowed for `eight_ball_profile_v1` (`name`, `dob`, optional `time`/`city`/`cc`/`tz`/`country`/`lat`/`lng`). No derived coordinate, catalog result, tier, credit, card prose, or payment data is stored. Added v0.50 (§5.E).

A zero-credit t3 top-up clears `eight_ball_pending_profile_v1` before opening the paywall so an abandoned Path A/B intent cannot auto-consume on return. No other change to either key's behavior.

Nothing else. No derived profile is stored — it's recomputed on each load. No analytics. No third-party endpoints. No third-party scripts. System fonts only.

**v0.50 allow-list amendment.** The "Nothing else" inventory above is superseded only by `eight_ball_saved_readings_v1` as specified in §5.E. The no-derived-profile rule remains literal: reopening an entry runs its stored input payload through the existing `profileFromPayload` → `buildProfile` and render path, so calculation outputs continue to have one authority.

## §5.B — User-initiated network calls (feedback v0.2.5, LS redirect v0.3.0 retired, Gumroad redirect v0.3.0.3)

The site permits exactly two user-initiated network calls:

1. **Feedback POST** — submission of a feedback form to the Netlify Forms endpoint at the same origin (added v0.2.5).
2. **Gumroad Buy Link redirect** — navigation away to the operator's hosted Gumroad checkout when the user taps the paywall CTA (added v0.3.0 as Lemon Squeezy path; swapped to Gumroad at v0.3.0.3).

Both calls are out-of-band — neither echoes user profile state. These are the only network calls permitted in the product runtime; everything else in §5 stands.

### Call 1 — Feedback POST

Constraints:

- **User-initiated only.** The submission fires only on user click of the explicit submit button. No timer, no auto-fire, no submission on page load, on shake, on result-render, or on any other implicit trigger.
- **User-authored content only.** The submitted payload contains exactly two fields: a free-text `message` and an optional free-text `contact`, both typed by the user inside the form. No localStorage profile data (`name`, `dob`, `time`, `country`, `lat`, `lng`, `city`, `cc`, `tz`) is included. No derived coordinates. No name, no DOB, no IP enrichment from the page, no analytics tags, no UTM params, no fingerprint. The form has no hidden field carrying user-data state.
- **Single named endpoint.** The `<form>` action is the same-origin Netlify Forms handler — either no `action` attribute (Netlify intercepts the POST and serves its branded thanks page) or `action="/?sent=1"` for a same-origin success redirect that the v0.2.5.2 banner JS detects and swaps to the in-page "thanks. read." state, then `replaceState` strips the query. The v0.2.5.2 success redirect is the shipped form (`tests/feedback_surface.test.js` locks `action="/?sent=1"`). No third-party form provider, no webhook fan-out, no secondary destination.
- **Native form post, no JavaScript fetch.** The form uses HTML `<form method="POST" data-netlify="true">` semantics. No `fetch()`, no `XMLHttpRequest`, no `navigator.sendBeacon`. The privacy scan (`tests/privacy_scan.test.js`) is unchanged and remains the enforcement.
- **Fail-silent.** On submission failure (offline, provider outage, network error), the browser handles the error natively. No retry queue, no remote error reporting, no telemetry of failure. If the user's browser is offline, the submit fails and the user can decide what to do.
- **Submission visibility.** Submissions land in the Netlify dashboard under the Forms tab. The operator may configure email notifications per Netlify defaults. Submissions are not echoed back to the page or persisted in localStorage.
- **No retention by 8ball.** 8ball-the-codebase does not retain submissions. Netlify-the-platform does (for as long as operator keeps them in the dashboard); that is operator's responsibility, not 8ball's runtime concern.

The about-modal copy is updated to disclose this surface honestly. Disclosure is the load-bearing piece: anyone reading the modal must learn that a feedback form exists and what happens when they submit it.

### Call 2 — Gumroad Buy Link redirect (v0.3.0 LS path → v0.3.0.3 Gumroad path)

Unlike the v0.3.0 Lemon Squeezy path, Gumroad does not accept a URL-encoded `success_url` query parameter on the Buy Link href; post-purchase redirect to `/?paid=t1` is handled entirely on Gumroad's side via the product's Content-tab "→ unlock your reads" Button (configured 2026-05-16 chat-27 directive F) which links to `https://the-eight-ball.netlify.app/?paid=t1`. The redirect mechanism is therefore single-source (Gumroad-side Content Button), not the belt-and-suspenders pattern of the prior LS path. Two-click post-purchase flow: paywall tap → Gumroad hosted checkout → Content-tab Button tap → `/?paid=t1`.

Gumroad processes payment via Stripe (Gumroad's internal Stripe Connect integration; operator's verified Connect account cleared 2026-05-16 per `~/Desktop/8ball/audits/gumroad_post_confirm_recheck_2026-05-16.md`). PayPal optional at buyer choice per Gumroad's default checkout. On successful payment, Gumroad displays the product Content page with the "→ unlock your reads" Button; on Button-tap, browser navigates to `https://the-eight-ball.netlify.app/?paid=t1`. Constraints:

- **User-initiated only.** Tap of the explicit paywall CTA. No timer, no auto-fire, no redirect on page load, on shake, on result-render, or on any other implicit trigger.
- **Hosted redirect, not in-page overlay.** Plain `<a href>` navigation. The Gumroad overlay/embed script is permanently rejected per §5's third-party-script ban — same constraint as the prior LS path, operator decision logged in the v0.3.0.3 cycle commit.
- **No user data leaves on the redirect.** The Buy Link URL is static — no profile fields encoded, no UTM tags, no tracking identifiers appended. The bare URL contains no query parameters at all (distinct from the v0.3.0 LS path which carried URL-encoded `checkout[success_url]`); `tests/payments_markup.test.js` v0.3.0.3 assertions lock this shape.
- **No SDK, no `fetch`, no webhook.** 8ball-the-codebase never talks to Gumroad programmatically; the redirect IS the entire payment surface.
- **Two-click post-purchase return (Gumroad-specific).** The post-purchase flow is two clicks (paywall → Gumroad checkout → Content-tab Button → `/?paid=t1`), structurally different from the LS path's single-click `checkout[success_url]` redirect. The Content-tab Button text is operator-controlled and ships as "→ unlock your reads" per chat-27 directive F.

## §5.E — Saved Readings (v0.50)

The result surface MAY offer `save reading` after a completed render. Saving is explicit and user-initiated; generating, rehydrating, sharing, opening the archive, or loading the page never saves an archive entry implicitly. The Previous Readings screen lists entries newest-first by `savedAt` and permits reopen, inline rename, confirmed individual deletion, and confirmed clear-all. `forget this device` clears both the current `eight_ball_profile_v1` payload and the complete Saved Readings archive.

**Storage minimum.** Each entry stores only `{ id, title, savedAt, profile }`. `profile` is a copy of the user-entered reconstruction payload allowed under §5; `id`, `title`, and `savedAt` exist only to identify, label, and order the local archive. Exact input profiles deduplicate to one saved entry. Derived coordinates, catalog/card output, card prose, entitlement tier, credits, tries, purchase metadata, and share artifacts are never copied into the archive.

**Revisit contract.** Opening a saved entry makes its profile the current local profile and recomputes the reading through the existing `profileFromPayload` / `buildProfile` / `showResult` path at the device's current entitled density. It does not call or alter the new-reading state machine, does not increment `eight_ball_tries_used_v1`, and does not decrement `eight_ball_credits_v1`. This is the same deterministic-rehydration class as cold-boot profile restoration and same-pair idempotence, not a new calculation or entitlement path. `core/`, `content/`, and `tests/fixtures.json` remain untouched.

**Failure and retention behavior.** Missing storage reads as an empty archive. Malformed outer JSON fails closed and is never silently overwritten by Save; the UI offers clear-all as recovery. Partially malformed arrays display only valid entries and disclose that some entries could not be read; the archive remains read-only until clear-all so an ordinary mutation cannot silently discard unreadable entries. Read, write, quota, or removal errors never crash generation or the calculation pipeline; the UI reports that saving/clearing is unavailable. Entries persist until the user deletes them, clears all, uses `forget this device`, clears site data, or the browser evicts local storage.

**Privacy boundary.** No account, backend, sync, analytics event, remote database, third-party storage, new endpoint, `fetch`, `XMLHttpRequest`, or beacon is introduced. The about modal and Previous Readings screen disclose the device-only scope and deletion controls. `tests/privacy_scan.test.js` allow-lists exactly `eight_ball_saved_readings_v1`; any second archive key or broader payload requires another §5 amendment.

## §5.F — Registry + Concordance (v0.51)

Concordance uses the Saved Readings archive as its only selector. Selecting two entry ids is transient UI state held in memory only. Comparison does not add a localStorage key, add a field to `eight_ball_saved_readings_v1`, create pair history, cache derived coordinates, store relation output, create a share artifact, or alter either saved entry. Closing or reloading the page discards the selection and result. The existing archive retention and deletion controls in §5.E remain the complete data-retention surface.

Both entries are validated by the existing archive loader and recomputed through the same calculation pipeline before comparison. Corrupt or partially valid archives preserve §5.E's fail-closed/read-only behavior; only valid displayed entries may be selected. If reconstruction or relation lookup fails, the UI reports that comparison is unavailable and leaves storage unchanged. Blocked/unavailable storage yields no selectable entries and does not affect ordinary reading generation. No account, backend, sync, analytics event, remote database, third-party storage, endpoint, network call, runtime dependency, or calculation-engine change is introduced. `tests/privacy_scan.test.js`'s key allow-list is unchanged from v0.50.

## §11. PII rule

Personal data of the operator, family, friends, or any other identifiable person is NEVER tracked content. The repo is private as of v0.2.0, but this rule is independent of repo visibility — tracked content is the durable boundary, not the current ACL state. Repos can flip; the rule survives the flip.

Specifically forbidden in any tracked file:

- The operator's full name in association with personal data (DOB, address, ID numbers).
- Test fixtures using real-person DOBs labeled to a real person.
- Any string matching combination patterns: `(operator-name) + (YYYY-MM-DD)` within 40 characters.
- SIRR cross-references outside of explicit boundary-naming doctrine.

The `tests/pii_scan.test.js` enforces a public-banned subset. The local audit at `audits/LOCAL_PII_AUDIT.md` closes the gap for personal data the public CI cannot know about.

**Fixture DOB rule (sub-rule of §11):** Fixture DOBs are chosen for the calc path they exercise, not anchored to any real person. If a calc path accidentally hits a real-person DOB, the fixture date is shifted by 12 years (preserves Chinese zodiac animal and life path mod-9 distribution while breaking real-DOB anchor) and the calc path is re-verified.

This file is **maintained by hand**, not generated. Update it when:

- A blocker is resolved or added
- A new tool/relay is automated (kill the old manual entry)

## §14. Operator calibration

Carried from an 8ball close-out session (2026-07-05), refined across a short review pass into fail-closed mechanisms rather than judgment calls. This section governs how the orchestrator (and any Claude lane) reads the operator and calibrates itself; it sits alongside §10's agent-role/lane architecture rather than replacing it.

**Objective gate (before the first file of any new cycle).**
1. Declare the objective in writing: money, mastery, or practice. One answer — it kills the others.
2. The slowest feedback loop that matters must resolve in <24h. If it doesn't, change the channel before writing code.
3. Name what gets deleted when this works. No subtraction plan = collection, not product.

**Reading the operator.**
- Reconstruct, don't clarify — but only for compression-trust: terseness that reads as deliberate shorthand on a reversible, low-blast-radius ask with an established acceptance criterion. Terseness that instead reads as unresolved ambiguity is not compression, whatever its length, and still gets a stated assumption or a clarifying question. Either way, an override-risk flag below still fires first.
- Override-risk flags, checked against the message, the current transcript, and canonical doctrine — not inferred tone or assumed fatigue, but not claimed as mechanically observable in isolation either. (a) and (b) require reading against standing law and thread history, not a keyword match; a close call fires the flag rather than waiting for certainty: (a) two asks in one message that contradict each other or a standing law; (b) scope doubling mid-thread without a new objective stated; (c) a directive that reverses a decision made <2h earlier in the same session; (d) a client-surfaced timestamp past midnight local on an irreversible-adjacent ask. Any flag fires → restate the interpretation and offer to defer to the morning read before executing. Supersedes an earlier draft of this rule keyed to "typos rising," which a chat transcript never actually exposes.
- Operational definitions, since these decide when (d) fires: "local" is the operator's own local clock, not server/UTC time; "irreversible-adjacent" means the ask sits within one step of something on the circuit-break list below (posts, merges, payments, deploys), not any midnight decision generally; "the morning read" is the operator's own re-read of the same thread after normal waking hours, not a fixed clock time.
- Irreversible-adjacent decisions flagged under (d) auto-defer to the morning read. Non-negotiable.
- Morning reads outrank midnight directives when the two conflict. Say so plainly — directness lands here, deference doesn't.
- Friction is designed, not narrated: the right action stays one tap; the wrong action requires a written sentence of justification. Naming a failure pattern is not the intervention.
- Taste calls (register, restraint, subtraction) get fast deference, even against conventional advice.

**Self-calibration (Claude, every lane).**
- Named failure mode: executing to avoid questioning — momentum reads as progress while direction drifts. The on-record instance is #69/#70 (2026-07-04): the same merge-before-audit-cleared-signal gap (L48) fired twice in one session, which is what made it a pattern rather than a slip.
- Weekly strategic check — "are we pointed at the right thing" — is exempt from terse-mode and is never requested; it fires on its own trigger. Mechanism, not memory: the newest `journal.md` entry carries a `next_strategic_read: YYYY-MM-DD` line; every session bootstrap reads it, and if it's overdue or missing the check runs before any task work, then the date rolls forward 7 days. Mirrors the §13 v0.43 self-firing rule-kill trigger exactly — a cadence a solo operator never watches cannot self-fire, so the trigger lives in a file that already gets read, not in memory.
- On unanimous-counsel overrides (all lanes agree, operator overrides anyway): the real objective must be stated out loud before executing. The override may be right, or may be revealing an unstated goal — lanes lack the operator's private context either way, so don't assume the override was correct going in. State the real objective, then judge the override against it.
- Own outcomes, not tasks.

**Standing laws.**
- Circuit-break on irreversibles: posts, merges, payments, deploys stay operator hands. Restates §10's controller-authority list; "posts" is the one item that list doesn't already name, since §10 predates the social-reach work.
- Dissent-then-BLOCK below a named evidence threshold (not dissent-then-execute). The threshold is a number or named condition pinned in writing for the specific decision at hand, never inferred per-session. If none is pinned, the absence is the answer: no threshold defined = below threshold, fail-closed rather than fail-open — same instinct as §12's gated-work discipline and the §13 amendment discipline, where an undefined gate is drift, not permission. Worked example already on file: the reach/build-B gate at `journal.md` 2026-06 ("build-B parked behind a NAMED evidence gate `[reach ≥ ~2–3k qualified] OR [≥1 strong divination-niche inflection signal]`") — not "N≥400," which is the *sub-threshold*, unmeasurable case logged the same week, not a gate value. Each gated decision pins its own number; none generalizes to a universal bar. This threshold rule governs empirical claims, gated-work decisions, and taste calls — it does not extend to safety, privacy, or violations of this doctrine's other standing rules. Those aren't waived by declaring an objective or invoking taste, no matter how firmly stated.
- Prototype before batch. Dry-run before install.
- ONE canonical file per domain, repo-resident, committed — no desktop vault, no split-truth layer. `DOCTRINE.md` is canon for product/process law, `journal.md` is canon for current state; this section is that file for operator calibration and is not re-pasted into chats as a parallel copy. Ephemeral review artifacts — audit briefs, relay run output, one-off critiques — can live outside the repo until their findings are reduced into canon; that's scratch space, not a second truth layer.
- Park = delete with a log line. WIP cap: 3.
- Audit intensity scales with blast radius. User-visible falsehoods outrank all polish.
- Refinement requires a named signal + hypothesis. No signal, no pass.
- One subtractive pass per cycle, scored by what it deletes.
- English only on public surfaces (western audience).
- No metric enters canon unless pinned from a dashboard. Estimates never get formatted as data.
- Weekly analytics read: file-backed like the strategic-read trigger above — a `next_analytics_read: YYYY-MM-DD` line in `journal.md`, checked at session bootstrap; overdue or missing = read runs now, then the date rolls forward 7 days. A calendar cadence nobody's watching doesn't self-fire — this section already made that argument once above; it applies here too.

**Build physics.**
- Generator-first: everything derivable derives from one data source; edit one line, regenerate all.
- Golden-master the full output space when it's deterministic — CI backbone, not a someday.
- Separate change-classes architecturally (copy / presentation / entitlement / engine) so cheap changes are audit-exempt by construction.
- When a constraint creates friction, reach for the primitive (worktree, script, file-drop) — never a new rule. Rules cost compliance forever; primitives don't.
- Agents communicate by file-drop from day one. Operator relays only where auth requires it.
- Post-merge close-out is a script, not a ritual.
- The distribution loop is one tap before any build automation exists.
- Automate nothing before three manual cycles of the same friction.

**Baseline.** This partnership's fluency starts at July-speed, not May-speed. Hold it there.
